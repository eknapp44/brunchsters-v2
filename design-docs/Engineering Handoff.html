What you're building

One paragraph, north-star clarity.

Brunchsters helps groups of friends actually do brunch instead of just saying "we should brunch." Hosts pick a spot (or several, with built-in voting), pick a time, invite the crew, and the app handles RSVPs, voting deadlines, nudges, and the inevitable schedule changes. Friend-tone, low-friction, mobile-first, but desktop works. Target: millennials + Gen Z, casual social planning.

Design system

Tokens, type, motion. Source of truth lives in styles/tokens.css.

Motion

Two easings: --ease (standard) and --ease-spring (delight moments). All animations respect prefers-reduced-motion — kills durations down to 0.001ms. Durations: 120ms (state changes), 200–250ms (transitions), 300–400ms (entry).

Key product decisions

The calls we made along the way. If something seems missing, this is why.

Two creation pathssimple vs. voting
Quick brunch (single-field conversational) is the default. The full 4-step wizard is reserved for when the host adds 2+ places or times.
Auth providersno email/SMS
Apple + Google only. Minimal friction. Name auto-grabbed; users drop straight into the dashboard with no profile-setup wall.
Tutorial styleno walls
The UI is the tutorial. No carousel, no coachmarks on first run. Empty states teach. The only coach is the FirstBrunchCoach fired after the user's first send.
Push permissiontiming matters
Deferred. Never asked at sign-up. Triggers after first RSVP yes or first send-invites — once we've earned it.
Invite link formatreadable + safe
brunchsters.app/b/{slug}-{token}. Expires 7 days after the brunch (post-event view stays accessible briefly). Hosts can revoke + regenerate.
Replan from past brunch"Do this again?"
v1 supported. Past brunch detail surfaces a "Do this again?" CTA that prefills title + location + invitee list into Quick Brunch and drops the user on the "when" step. Time stays blank — the user must pick. Banner ("Bringing back X — round two 🥂") persists across the flow; "Nah, start fresh" wipes prefill. Pure UI-layer plumbing in v1 — no schema changes needed; the create-brunch endpoint just receives a normal payload. The proactive morning-after surface ("How was Tartine?") remains v2.
Account deletionapp-store-grade
30-day soft delete. Cancels upcoming hosted brunches, withdraws pending RSVPs, keeps past brunches as historical ("former Brunchster"). Restore flow on sign-in within window.
RSVP statesricher than yes/no/maybe
Yes is the primary state and unlocks plus-ones, late, leave-early, dietary note. Maybe has "decide-by". No has an optional regret note + "invite me next time" opt-in.
Plus-onesopt-in per brunch
Off by default. Host opts in with a +1 / +2 / +3 cap. Plus-ones have an optional name (helps with reservations later).
Notificationstwo surfaces
App-wide preferences in Me tab (3 toggles: RSVPs / votes / reminders). Per-brunch mute in invitee detail footer (All on / Just big stuff / Muted).
Calendar integrationGoogle only for v1
Add-to-calendar chip inline on the time card. Disabled until time + place are locked — no tentative events, no ghost meetings. Helper line on the disabled chip: "Locks in once the vote closes." Outlook / Apple / ICS deferred.
Voting closuregraceful
Auto-close prompt at ≥66% participation. Tie at close → inline tie-breaker on the host's brunch detail (uses TieBreaker in EdgeStates.jsx). Host picks from the tied options in-place — no separate modal. No randomizer in v1.
Running late NEWtwo surfaces
Planning-ahead = the existing RSVP "late" state ("I'll arrive 15 min in"). Day-of = a one-tap action on the brunch detail screen within T-1h: "Tell the gang I'm running late" → fires a push to the group. Notification-style, no chat layer required.
Plus-one names editable NEWpost-submit
Invitees can edit plus-one names any time before brunch starts. Same surface as the original input. Edits silent (no notification — these are housekeeping changes).
Host transfer NEWin v1
From host detail → overflow menu → "Hand off hosting." Host picks any yes-RSVP'd invitee. New host gets a push + sees the brunch move into the "Hosting" lane on their dashboard. Old host stays as a regular attendee. One transfer per brunch — keeps the audit trail simple. New host inherits all permissions including cancellation.
Undo on cancel NEW5-second window
Cancellation modal "Call it off" closes → a 5-second undo toast appears before any notifications fire. If undone, the brunch returns to its prior state silently. If the 5s elapses, notifications go out and the cancellation is committed. Same pattern for "leave brunch" on invitee side.
Timezones NEWmulti-tz groups
Brunch time is authored in the host's local timezone and stored as UTC + an IANA zone string. Each invitee sees their local time, with a "(host's time: 11:00 PT)" subline only when the timezones differ. Voting time options follow the same rule. Calendar export uses the host's timezone.
Reduced-data states NEWloaded but empty
When a brunch loads with zero invitees / zero votes / zero RSVPs, show a tiny in-context line — not a full empty state. Voice example: "Nothing on the table yet 🍽", "Crickets so far — try a nudge?", "No votes in. Be the first?" See the Copy Variables section for the full set.
Quiet RSVP edgeT-24h, zero replies
Surface a QuietNudge card to the host: nudge / push the date / call it off. Never auto-cancel.
Cancellationhuman-toned
Confirm modal with optional reason field. Reason rides along on the cancellation notification — "Sam came down with something — let's reschedule."
Pop icon usagecurated, not blanket
Surgical by default — confirmed pills, success states, the "vote waiting" chip, hero metadata row. Toggle in Tweaks for the broad-rollout variant.

Empty state vs Skeleton — when to use which

Engineers were going to ask. Here's the rule.

Use Skeleton when

Data is loading.

We've fired a request and expect data back within ~2s. Shimmer placeholders match the eventual layout. Never shows under prefers-reduced-motion.

// fetchBrunches() pending
<DashboardSkeleton />

Use Empty state when

Data loaded — the set is genuinely zero.

Response returned, count is 0. Show EmptyDashboard-style illustrated state with a CTA. Has voice. Has personality.

// brunches.length === 0
<EmptyDashboard />

Reduced-data ≠ Empty. If a brunch loaded but has zero invitees or zero votes, that's reduced-data: show an inline playful line (see Copy Variables), not a full empty state.

Banned words: event, attendees, engage, community, action required, leverage, reach out. Reach-for: brunch on, round up the crew, the gang, on the table, call it off, Brunchster (canonical noun for an attendee — lean into it).

Voice rules that survived the copy pass

  • Errors stay in voice. Never An error occurred. Use "Couldn't reach the kitchen — try again?", "That brunch flew the coop" (404), "Something's burning — give it a sec". Tone is rueful, not apologetic.
  • "Brunchster" is the user noun. Use it in flavor copy, account flows, deletion ("former Brunchster"), and welcome states. Avoid in dense UI labels where it adds noise.
  • Pluralization is human. "1 friend" / "2 friends", "1 yes" / "3 yeses". Never "1 friend(s)". See Copy Variables for the full ICU plural rules.
  • Reduced-data lines are playful, not pitying. "Crickets so far" beats "No data available."

Things we didn't answer

The product calls engineering will hit. Don't guess — raise these.

Live vote updates · realtime mechanism. Voting cards animate when counts change. Polling vs WebSocket vs Server-Sent Events isn't designed for — design assumes "fast enough to feel live." Backend's call. aria-live regions are flagged in the a11y doc for when this lands.
Notification provider. Push perm prompt is designed but the actual provider (APNs + FCM directly? OneSignal? expo-notifications?) is undecided.
Email templates · transactional. Brand voice doc shows one example invite email. The full set (RSVP confirmation, reminder, cancellation, voting close, etc.) hasn't been written. Recommend a copy pass during email-template implementation, using the brand voice doc as floor.
Restaurant data source. Sample data is hard-coded. Real implementation needs a places API (Google Places? Foursquare? Yelp?). Search affordance is in the wizard but the data shape isn't speced.
Privacy: who sees whose RSVP? Currently invitees can see other invitees' RSVPs. Not yet decided whether "no" RSVPs are visible to the rest of the group, or just to the host.

Deferred to v2 (do not build)

Things we considered and explicitly punted. Don't sneak these in.

Chat / social layer

Threading, reactions, "Sam liked your message." Deferred — Brunchsters is a planner, not a chat app.

Favorite places / friends

User-curated lists. Suggestions in the wizard are sample data for now.

Calendar connect

Google / Apple / Outlook OAuth + free-busy overlay on the time picker. Considered for the empty-dashboard onboarding (see EmptyDashboard.v2-reference.jsx) and deferred. v1 has no calendar awareness — host picks times blind, invitees see times in their local TZ but nothing checks conflicts.

Reservations

OpenTable / Resy integration. Plus-one head count is designed with this in mind.

Recurring brunches

"Every other Sunday with Maya." Not in v1.

Brunch photos

Past detail has a placeholder. Avatar upload is in scope; brunch photos aren't.

Morning-after loop

"How was Tartine?" prompt + plan-again push. The "Do this again?" CTA on past detail IS in v1 (prefills location + invitees into Quick Brunch, time stays blank) — what's deferred is the proactive morning-after surface that asks how it went and nudges replanning.

Mobile-native detail screens

v1 mobile shell uses the responsive desktop components for host / invitee / past detail — they re-flow OK at 420px but aren't laid out for the form factor. Mobile dashboard, quick-brunch sheet, and the action bar on past detail ARE bespoke. Full mobile-native detail redesigns (sticky action bar, swipe gestures on attendee list, tap-to-text-host) are v2.

i18n / RTL

US-English only. Date / time formats are en-US baked in.

Offline / queueing

Action queue when offline. Currently assumes online; the toast system covers transient failures.

Group-aware restaurant picks

"Best for this group" algorithmic suggestion. Search UI is generic for v1.

Re-invite from "No"

"Maya said no but tagged 'invite me next time' — bring her into the next one?" Host-side surface. Data flag is captured in v1; the UI is v2.
